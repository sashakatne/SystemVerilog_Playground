Review findings:
  - The current busy output is not a clean synchronous operation-complete
    handshake. DataPath drives busy as busy_ff | set_busy, while the FSM drives
    set_busy directly from raw zero in Init. That can create a visible busy
    pulse before the zero operation is actually held in Load/Write.
  - The waveform shows a later zero request arriving while the first sweep is
    still in Write. This explains why the testbench can report completion early
    and then start the next transaction too soon.
  - The high/low address stores are transparent latches. If addr changes while
    ld_high or ld_low is high, the bounds can move during the load window.
