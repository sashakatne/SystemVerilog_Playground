Known issues (NOT fixed in this commit):
  - TC3 (write + zero same range): after the busy handshake the
    debug-trace shows R1=0xFF, R2=0x00, counter ends at 0xFE (not 0x00 as
    the FSM math suggests it should). memory[0x55] retains its pre-zeroing
    value 0xAA. The handshake fix rules out the original timing race, so
    the residual failure points at either a Questa scheduling subtlety in
    the latch+JKFF+state-FF mix or an FSM bug that I could not isolate in
    the time budget.
  - TC4 (write during zeroing): the DUT uses `we = write | zero_we`, so a
    user write during the Load state (when addr_sel is still 0) lands at
    the user-requested address with the user-requested data, before the
    Write state's addr_sel takes effect. Either the DUT needs to gate
    `write` when busy is 1, or the TB expectation must be relaxed.
  - TC6 (busy within 1 cycle of zero): the BusyFF (JKFF in DataPath)
    samples set_busy/clr_busy from the OLD state at each posedge, so the
    busy flag rises 2 cycles after zero=1, not 1. The TB's
    `wait_clock_cycles(1); if (!busy) ...` check fires too early. The
    DUT-side fix would be either to assert set_busy in Init when zero=1
    (combinationally) or to bypass the JKFF and drive busy from state.

Caveats:
